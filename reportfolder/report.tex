\chapter*{ABSTRACT}
\addcontentsline{toc}{chapter}{ABSTRACT}
\phantomsection

\begin{small}
This project develops a multi-agent data analysis pipeline that turns raw CSV-based financial and sales data into structured insights and charts. The system is designed around a shared state model and a planned sequence of six specialized agents: structural profiling, semantic tagging, preprocessing, statistical analysis, validation, and report generation. Each agent performs one focused responsibility and passes its results to the next stage through a controlled pipeline.

At the mid-point of development, the first four agents are implemented and the backend pipeline runs end to end from ingestion through statistical analysis. The codebase now includes robust CSV and Excel loading, semantic metadata generation, deterministic preprocessing, and statistical analysis with chart generation. Output validation (Agent~5) and report generation (Agent~6) are not yet implemented and form the main remaining work, together with refinement, testing, and evaluation on more datasets. The project is moving toward a practical analytics tool that can reduce manual effort and make technical analysis more accessible to non-technical users.

\end{small}

\vspace{1em}
\noindent\textbf{Keywords:} Multi-agent systems, data analysis, financial reporting, sales analytics, LangGraph, preprocessing, validation, automated reporting

\chapter{INTRODUCTION}

\section{Background}

Data-driven decision-making has become a foundational requirement for organizational effectiveness in health, agriculture, education, and commerce. In Nepal, the government's Digital Nepal Framework explicitly prioritizes data-centric governance across these sectors. However, the operational reality for most organizations particularly those outside the Kathmandu Valley remains characterized by manual spreadsheet management and underutilized datasets. The technical barrier to entry for modern analytics is high: tools demand proficiency in query languages, schema design, or statistical software, while enterprise platforms carry licensing costs that are unsustainable for donor-funded non-governmental organizations (NGOs) and municipal bodies. The emergence of LLMs and agentic orchestration frameworks has created a new design space for autonomous software systems \cite{dwivedi2023}. LLMs can now interpret column semantics and narrate statistical findings in natural language, while agentic frameworks such as LangGraph enable reliable, stateful, multi-step automation. The convergence of these technologies makes it feasible to build a pipeline that replicates the workflow of a human data analyst validation, cleaning, profiling, visualization, statistical testing, interpretation, and reporting without requiring human intervention at each step.

\section{Motivation}

The motivation for this project is based on the following points:

\begin{itemize}
	\item Despite the availability of data, many organizations lack the technical expertise required to analyze and interpret it effectively.
	\item Traditional data analysis tools often require statistical knowledge, programming skills, or manual report interpretation.
	\item Existing AI-based systems using LLMs may generate unverified or hallucinated insights, leading to unreliable conclusions.
	\item Recent advances in Artificial Intelligence and Multi-Agent Systems provide an opportunity to automate complex analytical tasks.
\end{itemize}

\section{Problem Definition}

Organizations accumulate data daily sales records, transaction logs, expense sheets, customer information yet cannot efficiently convert this data into insights that drive strategy and decision-making.

Existing challenges:

\begin{enumerate}
	\item \textbf{Manual workflows:} Data analysis currently requires skilled staff to manually clean data, calculate metrics, build charts, and compile reports a process taking days to weeks

	\item \textbf{Lack of standardization:} Without a systematic methodology, different people produce inconsistent results, making it difficult to compare periods or validate findings

	\item \textbf{Limited accessibility:} Analytics tools require specialized knowledge. Non-technical stakeholders cannot ask ad-hoc questions without analyst intermediaries

\end{enumerate}

This project proposes an end-to-end automated system that takes raw sales and financial CSV data and produces validated, auditable, professionally formatted reports with trend insights and recommendations.

\section{Objectives}

The primary objectives of this project are:

\begin{enumerate}
	\item Develop an autonomous multi-agent system that converts raw sales and financial data (CSV format) into professional analytical reports and forecasts without manual coding

	\item Democratize basic financial analytics and sales optimization for small to medium businesses and organizations through a no-code web interface
\end{enumerate}

\section{Scope and Applications}

\hspace*{\parindent}\textbf{Technical Scope:}

\begin{itemize}
	\item Supports common business data types: sales records, transaction logs, expense sheets, in CSV format
	\item A six-agent pipeline design; four agents (profiling, semantic tagging, preprocessing, and statistical analysis) are currently implemented, while validation and report generation are planned
	\item Runs as web application without requiring to download additional software
\end{itemize}

	extbf{Primary Use Cases:}

\begin{enumerate}
	\item \textbf{Sales Performance Tracking:} Automated analysis of sales data to identify trends, top-performing products, seasonal patterns, and revenue forecasts for planning

	\item \textbf{Financial Reporting \& Budgeting:} Generation of expense summaries, income vs. expenditure analysis, budget tracking, and monthly/quarterly financial reports for small to medium organizations

	\item \textbf{Retail \& E-commerce Analytics:} Customer purchase behavior, inventory turnover analysis, demand forecasting, and sales channel comparison

	\item \textbf{Small Business Operations:} Cash flow trends, profit margin analysis, supplier/payment tracking, and operational dashboards without requiring accounting software expertise

	\item \textbf{Financial Data for Non-Finance Teams:} Simplified reports that translate raw financial/sales data into plain-language insights for managers, project leads, and decision-makers
\end{enumerate}

\chapter{LITERATURE REVIEW}

\section{Traditional Data Analytics Platforms}
Platforms for Business Intelligence like Microsoft Power BI, Tableau, and Google Looker Studio are popular choices when it comes to data visualization, creation of dashboards, and decision making. These technologies offer a variety of data connectivity options, visualization functions, and reporting features, helping companies track their key performance indicators and business processes. With all their ecosystem and integration possibilities, these BI systems can be used by large corporations. However, there is a number of disadvantages of BI systems as well. First of all, users need to know database schema, configure their dashboards, and prepare the data themselves. Moreover, licensing and other infrastructure expenses may prove to be expensive for small companies, educational institutions, and non-profit organizations lacking funds. However, one of the major drawbacks of traditional BI systems is their lack of automation in interpretation, relying only on visualization.
According to Dwivedi et al. \cite{dwivedi2023}, large language models are changing data analytics. LLMs let us interact naturally, reason automatically, and get smart decision support. The authors argue that we need to progress from static dashboards to AI systems that offer insightful auto-recommendations. This points out an important research gap. To address it, we should develop intelligent, multi-agent data analyst systems.

\section{AI-Assisted Analytics and Large Language Model-Based Data Analysis}
LLMs are changing data analytics with their ability to chat naturally about data \cite{dwivedi2023}. Tools like ChatGPT with Code Interpreter and other assistant-style systems let people upload datasets and get back code, charts, stats analyses, and plain text explanations \cite{openai2024}. This is big because it makes data analysis easier for those without a ton of technical skills.

One huge plus is how easy these platforms are to use. Regular folks can talk to a dataset without needing loads of coding knowledge. Researchers show that LLMs can handle lots of jobs, like cleaning data, doing exploratory analysis, spitting out visuals, and even putting together reports \cite{sun2024}.

Still, there are some downsides. For one, getting the same result each time is tricky since outcomes rely heavily on the questions you ask. You might phrase things differently and end up with entirely distinct findings. Also, these systems tend to be single agent setups that don’t follow a strict workflow or include checks for accuracy. There’s no built-in QA process, and sometimes the code generated can have errors or missing steps \cite{liu2023chatgpt, maynez2020faithfulness}. This really drives the need for more complex systems that employ different agents for doing analysis, verifying results, and generating reports.

\section{Automated Data Visualization from Tabular/CSV Data}
Several studies have investigated automated visualization generation from structured datasets. Data2Vis \cite{dibia2019} proposed a sequence-to-sequence neural architecture that translates datasets into Vega-Lite visualization specifications. VizML \cite{hu2019} applied machine learning to learn visualization design choices from over one million dataset-visualization pairs. Draco \cite{moritz2018} formalized visualization design knowledge as logical constraints to recommend effective visual encodings. DeepEye \cite{luo2018} further automated chart generation and ranking. These studies collectively demonstrate the feasibility of automated visualization generation and provide strong theoretical support for the Visualization Agent proposed in this research.

The main insight from this line of work is that visualization creation can be treated as a recommendation or generation task rather than a purely manual design task. This is especially relevant for CSV-based analysis workflows, where the system must decide whether to use bar charts, line charts, scatter plots, or other encodings based on the structure and semantics of the tabular data. The proposed project adopts this idea by using an agent that converts cleaned tabular outputs into appropriate static visualizations for reports.

\section{Agentic AI Frameworks and Multi-Agent Systems}
The newest stuff in Agentic AI includes LangGraph, CrewAI, and AutoGen frameworks that help different AI agents work together on big projects \cite{langchain2024, wu2023}. Traditionally, one AI does all the work, but these new setups let multiple agents take on tasks based on what they’re best at.

LangGraph helps manage agent teamwork with things like conditional routing and checkpointing. Meanwhile, AutoGen is all about letting agents chat back and forth, and CrewAI handles task delegation based on roles. Altogether, these tools show promise for building really smart, self-reliant systems that can figure out tough problems, plan, and coordinate tasks efficiently.

According to Wu et al. \cite{wu2023}, when you set up proper structures for AI conversations, teams of specialized agents can ace complex assignments. This covers everything from coding and math to answering questions and making decisions. It’s clear that such collaborations do better than lone agents.

Still, most work right now only applies these frameworks to areas like software engineering, web searches, Q\&A, and general reasoning. Not many have looked into how well they fit into fixed data analysis processes that mix deterministic stats ops with more uncertain language-model thinking. This gap presents an opportunity for the development of multi-agent systems specifically designed for automated data analysis and business intelligence workflows.

\section{Existing Research on Automated Data Analysis Systems}
Researchers are looking into artificial intelligence to make data analytics easier. They’re mixing machine learning, big language models, and natural language processing to help with exploring data, making it visual, and getting insights.

Sun et al. came up with LAMBDA, a Large Model Based Data Agent, which uses special AI agents to do data analysis \cite{sun2024}. These agents follow instructions in regular language to do tasks like code generation, stats, and visualizations. The cool thing is, when these agents work together, they improve accuracy and dependability more than solo agents \cite{sun2024}.

While there's been progress, most systems now only cover certain parts of the analytics job. For instance, statistical tools are great but aren’t very user-friendly, and some language model tools don’t always deliver consistent results because they rely too much on how you write prompts. Also, many current tools work in isolation they don’t all come together to form that full analytics process which would include everything from pre-processing data to generating reports.

So, we need a better way. There’s a call for a complete multi-agent setup that provides reliable and repeatable data analytics. This could revolutionize how we handle big data tasks.

\section{Research Gap and Project Positioning}
According to the literature reviewed, there is a lack of studies dedicated to multi-agent data processing systems that combine deterministic analytical algorithms with LLM-based reasoning in a reliable end-to-end analytics pipeline \cite{sun2024, wu2023}. Traditional BI tools require significant user involvement and technical expertise \cite{dwivedi2023}. While LLM-based systems improve accessibility, they do not inherently guarantee reproducibility, auditability, and structured execution workflows \cite{liu2023chatgpt, maynez2020faithfulness}. Multi-agent frameworks provide orchestration capabilities but have primarily been applied to software engineering and general reasoning tasks rather than autonomous data analytics and reporting \cite{wu2023, langchain2024}.
Therefore, this project proposes a LangGraph-based multi-agent analytics system in which specialized agents perform data ingestion, preprocessing, analysis, visualization, validation, and report generation. Deterministic analytical methods will be used for statistical computations, while an LLM-based agent will generate insights and explanations. The proposed system aims to improve reproducibility, accessibility, and automation of data analytics workflows for educational institutions, small businesses, and non-technical users.

\chapter{THEORETICAL APPROACH}

The implementation in this project depends on two theoretical foundations: the internal mechanics of Transformer-based Large Language Models (LLMs), which power the semantic tagging stage, and the local inference pipeline provided by Ollama, which is treated as the official LLM backend of the system. This chapter explains only the concepts that are directly required to understand how the project generates schema metadata and grounded narrative text, keeping the treatment implementation-oriented rather than exhaustive.

\section{Large Language Model Theory}

A Large Language Model is a neural network trained to model the probability distribution of natural language, so that given a sequence of previous tokens it can estimate the most likely next token. In this project the LLM is not used to write free-form prose only; it is used as a structured reasoning component that reads column metadata and returns a strict JSON schema blueprint. Understanding how the model turns text into predictions is therefore necessary to explain why the semantic tagging agent is reliable enough to drive deterministic preprocessing.

\subsection{From Transformers to Modern LLMs}
Early sequence models such as recurrent neural networks (RNNs) and long short-term memory (LSTM) networks processed tokens one step at a time. This sequential dependency made training slow and made it difficult to capture relationships between tokens that are far apart in a sentence. The Transformer architecture removed recurrence entirely and replaced it with a self-attention mechanism that relates every token to every other token in parallel. Because all positions are processed simultaneously, Transformers train efficiently on modern hardware and model long-range dependencies directly. Modern LLMs such as the Llama family are scaled Transformer networks trained on large text corpora, and they are the class of models executed locally through Ollama in this project.

\subsection{Decoder-Only Transformer Architecture}
The models used for text generation, including Llama, adopt a \emph{decoder-only} Transformer design. The network is a stack of identical decoder blocks, each containing a masked multi-head self-attention sub-layer and a position-wise feed-forward sub-layer. A causal mask ensures that the prediction for a token at position $t$ can only attend to tokens at positions $1$ to $t$, never to future tokens. This left-to-right constraint is what makes the model autoregressive and suitable for generating one token at a time.

\subsection{Tokenization and Vocabulary}
Before any computation, raw text is split into discrete units called tokens using a subword tokenizer (typically byte-pair encoding). Each unique token maps to an integer index in a fixed \emph{vocabulary} of size $V$. Subword tokenization keeps the vocabulary compact while still being able to represent rare or unseen words as combinations of smaller units. In the project, the prompt sent to the model, which contains column names, inferred types, and sample values, is tokenized in exactly this way before inference.

\subsection{Embedding Layer}
Each token index is converted into a dense vector using a learned embedding matrix $E \in \mathbb{R}^{V \times d}$, where $d$ is the model dimension:
\begin{equation}
x = E(\text{token})
\end{equation}
Here $\text{token}$ is the integer index of the token, $E$ is the embedding lookup table, and $x \in \mathbb{R}^{d}$ is the resulting embedding vector. This vector is the numerical representation of the token that the rest of the network operates on.

\subsection{Positional Encoding}
Because self-attention is order-agnostic, positional information must be injected explicitly. Sinusoidal positional encodings add a position-dependent signal to each embedding:
\begin{equation}
PE_{(pos,\,2i)} = \sin\!\left(\frac{pos}{10000^{2i/d}}\right)
\end{equation}
\begin{equation}
PE_{(pos,\,2i+1)} = \cos\!\left(\frac{pos}{10000^{2i/d}}\right)
\end{equation}
where $pos$ is the position of the token in the sequence, $i$ is the dimension index, and $d$ is the model dimension. Even dimensions use a sine function and odd dimensions use a cosine function. The encoding is added to the token embedding so that the model can distinguish tokens by their position as well as their identity.

\subsection{Self-Attention and Multi-Head Attention}
Self-attention lets each token gather information from other tokens. From the input, three projections are formed: queries $Q$, keys $K$, and values $V$. The scaled dot-product attention is defined as:
\begin{equation}
\text{Attention}(Q,K,V) = \text{Softmax}\!\left(\frac{QK^{\top}}{\sqrt{d_k}}\right)V
\end{equation}
Here $Q$, $K$, and $V$ are matrices of query, key, and value vectors, $d_k$ is the dimension of the key vectors, and $\sqrt{d_k}$ is a scaling factor that prevents the dot products from becoming too large and destabilising the softmax. The term $QK^{\top}$ measures the similarity between each query and every key; the softmax converts these similarities into weights that are used to form a weighted sum of the value vectors.

Rather than performing attention once, the Transformer uses several parallel attention \emph{heads}, each with its own projections, and concatenates their outputs:
\begin{equation}
\text{MultiHead}(Q,K,V) = \text{Concat}(\text{head}_1, \dots, \text{head}_h)\,W^{O}
\end{equation}
where each $\text{head}_i = \text{Attention}(QW_i^{Q}, KW_i^{K}, VW_i^{V})$, $h$ is the number of heads, $W_i^{Q}, W_i^{K}, W_i^{V}$ are per-head projection matrices, and $W^{O}$ is the output projection. Multiple heads allow the model to attend to different types of relationships simultaneously.

\subsection{Feed-Forward Network}
Each decoder block applies a position-wise feed-forward network (FFN) independently to every token vector:
\begin{equation}
\text{FFN}(x) = W_2\big(\text{ReLU}(W_1 x + b_1)\big) + b_2
\end{equation}
where $x$ is the token representation, $W_1$ and $W_2$ are weight matrices, $b_1$ and $b_2$ are bias vectors, and $\text{ReLU}$ is the rectified linear activation. The FFN expands the representation to a higher dimension, applies a non-linearity, and projects it back, giving the model additional capacity to transform each token.

\subsection{Residual Connections and Layer Normalization}
To train deep stacks stably, each sub-layer output is added back to its input through a residual connection and then normalized:
\begin{equation}
\text{LayerNorm}\big(x + \text{Attention}(x)\big)
\end{equation}
The residual term $x$ preserves the original signal and helps gradients flow through many layers, while layer normalization rescales the summed vector to have a stable mean and variance:
\begin{equation}
\text{LayerNorm}(u) = \gamma\,\frac{u - \mu}{\sqrt{\sigma^2 + \epsilon}} + \beta
\end{equation}
where $u$ is the input vector, $\mu$ and $\sigma^2$ are its mean and variance, $\epsilon$ is a small constant for numerical stability, and $\gamma$ and $\beta$ are learned scale and shift parameters.

\subsection{Logits, Softmax, and Next-Token Prediction}
After the final decoder block, a linear projection maps the last token representation to a vector of \emph{logits} $z$ of length $V$, one score per vocabulary token. A softmax converts these logits into a probability distribution:
\begin{equation}
P(y_i) = \frac{\exp(z_i)}{\sum_j \exp(z_j)}
\end{equation}
where $z_i$ is the logit for token $i$ and the denominator sums over all vocabulary tokens $j$. The model then predicts the next token conditioned on all previous tokens:
\begin{equation}
P(x_t \mid x_{<t})
\end{equation}
where $x_t$ is the token being predicted and $x_{<t}$ denotes all tokens before position $t$. Text is generated \emph{autoregressively}: the predicted token is appended to the sequence and fed back in to predict the following token, repeating until a stopping condition is reached.

During the model's original training, these predictions are optimised with the cross-entropy loss:
\begin{equation}
L = -\sum_i y_i \log(p_i)
\end{equation}
where $y_i$ is $1$ for the correct next token and $0$ otherwise, and $p_i$ is the predicted probability for token $i$. This objective drives the probability mass toward the correct token. The project itself performs inference only and does not train the model, but the loss explains why the pre-trained model assigns high probability to coherent, well-formed output such as the valid JSON schema the tagging agent depends on.

\section{Ollama Model and Inference Architecture}

Ollama is the official LLM backend of this system. It is a lightweight runtime that loads and serves open-source Transformer models, such as the Llama family, entirely on the local machine and exposes them through a local REST API. Rather than viewing Ollama merely as a piece of software, this section describes the full Transformer inference pipeline that Ollama executes when the semantic tagging agent sends it a prompt.

\subsection{Why Ollama Is Used}
The project uses Ollama because it performs inference locally, which gives the system four practical benefits. First, \emph{privacy}: the dataset-derived prompts never leave the machine, which matters for financial and sales data. Second, \emph{offline execution}: the pipeline can run without internet access once the model is pulled. Third, \emph{low and predictable latency and cost}: there are no per-request API charges or network round-trips to a cloud provider. Fourth, \emph{easy deployment}: a single runtime manages model loading and serving, which simplifies reproducing the system. These properties make Ollama an appropriate production backend for an analytics tool intended for small organizations.

\subsection{Local Model Loading and REST API}
When a model such as Llama~3.x is requested, Ollama loads the model weights from local storage into CPU or GPU memory and keeps them resident so repeated requests avoid reload overhead. Inference can run on CPU or be accelerated on a GPU, and memory usage is dominated by the model weights plus the attention cache described below. Agent~2 communicates with the running model through Ollama's local REST API: it issues a chat-completion request containing a system instruction and a user message, and receives the generated text in response. From the application's perspective this is a simple local HTTP call, while all the Transformer computation happens inside the Ollama process.

\subsection{Prompt Processing Pipeline}
Inference proceeds through the same Transformer stages described in the previous section, executed in the forward direction only.

\textbf{Tokenization.} The incoming prompt is split into subword tokens and mapped to integer indices in the model's vocabulary.

\textbf{Embedding generation.} Each token index is converted into a vector through the embedding matrix, $x = E(\text{token})$, and positional information is added using the sinusoidal encoding $PE_{(pos,2i)} = \sin(pos/10000^{2i/d})$ and $PE_{(pos,2i+1)} = \cos(pos/10000^{2i/d})$.

\textbf{Decoder stack and multi-head attention.} The embedded sequence passes through the stack of decoder blocks. In each block, masked multi-head self-attention computes $\text{Attention}(Q,K,V) = \text{Softmax}(QK^{\top}/\sqrt{d_k})V$, letting each token attend to all earlier tokens in the prompt.

\textbf{Feed-forward network.} Each block then applies $\text{FFN}(x) = W_2(\text{ReLU}(W_1 x + b_1)) + b_2$ to every token position.

\textbf{Residual connections and layer normalization.} Every sub-layer is wrapped with a residual connection and normalization, $\text{LayerNorm}(x + \text{Attention}(x))$, keeping the deep computation numerically stable during inference.

\textbf{Logits, softmax, and token sampling.} After the final block, the last position is projected to logits over the vocabulary and normalized with the softmax $P(y_i) = \exp(z_i)/\sum_j \exp(z_j)$, producing the next-token distribution $P(x_t \mid x_{<t})$. A sampling strategy (for example temperature-controlled sampling) selects the next token from this distribution. The project sets a low temperature so that the output is nearly deterministic, which is important because the response must be strictly valid JSON.

\textbf{KV cache.} To generate efficiently, Ollama stores the key and value tensors computed for previous tokens in a \emph{KV cache}. When the next token is produced, only the new token's query must attend to the cached keys and values, so the model does not recompute attention over the entire sequence at every step. This is the main optimization that makes autoregressive local generation fast.

\textbf{Autoregressive generation and response decoding.} The sampled token is appended to the sequence and fed back into the model to predict the following token. This loop continues until an end-of-sequence token or length limit is reached. The generated token indices are finally decoded back into text and returned over the REST API to Agent~2, which parses the JSON schema blueprint from it.

\section{Shared State Design}
The shared state contains the input path, the raw profile, the semantic blueprint, the cleaned dataset, the scaling parameters, the preprocessing log, the quality score, the statistics, the chart paths, an error list, and reliability metadata. This design allows each agent to focus on one responsibility while still keeping the whole workflow connected. Fields for the validation result and the final report path will be added to the state once Agent~5 and Agent~6 are implemented.

The state also carries reliability metadata. Each stage can contribute a confidence score and evidence list, which helps the system decide whether a later stage should proceed. That is useful in the report because it shows that the project is not just generating outputs, but also measuring the reliability of those outputs.

\section{Methodology}
The methodology combines deterministic data processing with LLM-assisted semantic reasoning. Structural checks, preprocessing rules, and statistics are implemented in Python, while the validation and reporting stages remain planned work. Semantic interpretation uses a local LLM served through Ollama, and the planned final narrative generation will use the same backend; both stages are backed by fallback logic so the pipeline remains usable when the model is unavailable.

Methodologically, the project follows a layered analytics model. The early stages reduce raw data into a trustworthy form, the middle stages compute measurable patterns, and the later stages turn those patterns into human-readable findings. That mirrors the workflow of a human analyst while still keeping the process automated and repeatable.

The implementation approach is also incremental. Each agent was developed as a distinct module, which made it easier to test one stage at a time and identify integration issues early. This is a good fit for a mid-term report because it shows progress in the structure of the system, not only in the final output.

\chapter{IMPLEMENTATION DETAILS}

\section{Agent 1: Structural Profiler}

\textbf{Purpose.} Agent~1 (\texttt{agent1\_structural\_profiler}) reads the input file and records its structure. It deliberately performs no fixing and no inference; it only observes and records, giving later stages a factual baseline. The overall workflow is shown in Figure~\ref{fig:agent1}.

\textbf{Inputs.} It reads \texttt{csv\_path} from the shared \texttt{GraphState}, an optional \texttt{sheet\_name} for Excel inputs, and the running \texttt{errors} list.

\begin{figure}[H]
\centering
\includegraphics[width=0.6\textwidth,keepaspectratio]{agent1pic.png}
\caption{Workflow of Agent~1: Structural Profiling}
\label{fig:agent1}
\end{figure}

\textbf{File loading.} Loading is dispatched by extension in \texttt{\_load\_dataframe}. Excel files are read by \texttt{\_read\_excel\_file}, which reads a pinned sheet directly or concatenates all sheets (outer join) while tagging each row with a \texttt{\_source\_sheet} column and warning on schema mismatch. CSV files are read by \texttt{\_read\_mixed\_delimiter\_csv}, which reads raw lines through \texttt{\_read\_csv\_lines} using the encodings \texttt{utf-8-sig}, \texttt{cp1252}, and \texttt{latin-1} in turn.

That reader detects the delimiter per line by comparing comma and semicolon counts, rebuilds each row with Python's \texttt{csv} module, and retries with the alternate delimiter when the field count does not match the header. Unknown extensions fall back to CSV parsing.

\textbf{Structural profiling.} Per column it records the Pandas \texttt{dtype}, missing count (\texttt{isna().sum()}), missing rate, unique count (\texttt{nunique(dropna=False)}), and the first three non-null \texttt{sample\_values}. Dataset-level fields include shape, overall missing rate, and duplicate rows from \texttt{df.duplicated().sum()} with the duplicate rate.

\textbf{Reliability.} \texttt{\_compute\_agent1\_confidence} produces a confidence value that decreases with missingness and duplication:
\begin{equation}
c_1 = \text{clamp}\!\left(1 - 0.5\,\frac{m}{100} - 0.3\,\frac{d}{100},\; 0,\; 1\right)
\end{equation}
where $m$ is the overall missing rate and $d$ the duplicate rate (percentages). The stage is marked \texttt{ready} when $c_1 \geq 0.8$ and \texttt{needs\_review} otherwise, via \texttt{update\_reliability}.

\textbf{Output and interaction.} On success it writes \texttt{raw\_profile}, caches the DataFrame in \texttt{\_df\_cache}, and updates \texttt{errors} and \texttt{reliability}. If loading fails, the exception is appended to \texttt{errors} and the state returns early. The cached DataFrame and profile are consumed directly by Agent~2.

\section{Agent 2: Semantic Tagger}

\textbf{Purpose.} Agent~2 (\texttt{agent2\_semantic\_tagger}) produces the \texttt{schema\_blueprint}, a per-column description of intended type, semantic role, and cleaning policy. When the LLM is invoked, the model is served locally through Ollama running a Llama-family Transformer. Figure~\ref{fig:agent2} summarizes the flow.

\textbf{Inputs.} It uses \texttt{raw\_profile} and the cached DataFrame \texttt{\_df\_cache} from Agent~1, plus the \texttt{errors} list. If no DataFrame is present, it records an error and returns.

\begin{figure}[H]
\centering
\includegraphics[width=0.6\textwidth,keepaspectratio]{agent2pic.png}
\caption{Workflow of Agent~2: Semantic Tagging}
\label{fig:agent2}
\end{figure}

\textbf{Type sniffing.} Before any model call, \texttt{\_infer\_intended\_types} performs cheap local inference, mapping each column to \texttt{numeric}, \texttt{datetime}, \texttt{boolean}, \texttt{string}, or \texttt{unknown} using the dtype and an 80\% parseability threshold for object columns.

\textbf{Prompt construction and inference.} \texttt{\_build\_llm\_prompt} builds a minimal prompt with each column's name, inferred type, missing rate, unique count, and three samples. A fixed system prompt requires the model to return \emph{only} valid JSON describing \texttt{intended\_type}, \texttt{semantic\_tag}, \texttt{is\_identifier}, \texttt{scaling\_allowed}, \texttt{imputation\_strategy}, \texttt{encoding\_strategy}, and \texttt{null\_policy}, at a low temperature (0.1) for near-deterministic output.

When the schema exceeds \texttt{LLM\_SINGLE\_CALL\_THRESHOLD} (20) columns, \texttt{\_split\_columns\_into\_batches} splits it into batches of \texttt{LLM\_BATCH\_SIZE} (15). \texttt{\_call\_llm\_for\_schema\_blueprint\_with\_retry} calls the model per batch and, if a truncated response fails JSON parsing, recursively halves the batch, retries, and merges results with \texttt{\_merge\_schema\_blueprints}. \texttt{\_parse\_schema\_blueprint\_response} accepts raw or fenced JSON and, as a last resort, extracts the outermost \texttt{\{...\}} span.

\textbf{Enrichment and confidence.} \texttt{\_apply\_missingness\_policy} and \texttt{\_enrich\_missingness\_metadata} fill any missing \texttt{semantic\_tag} via \texttt{\_infer\_semantic\_tag\_from\_metadata} and attach a \texttt{column\_assessment}, \texttt{null\_policy}, and \texttt{encoding\_strategy}. Columns above \texttt{MISSINGNESS\_ANALYSIS\_THRESHOLD\_PCT} (20\%) are marked \texttt{analysis\_allowed = false}. \texttt{\_calculate\_semantic\_confidence} scores each tag from 0--100, and a \texttt{\_\_metadata\_\_} block records the risk assessment and preprocessing recommendation.

\textbf{Fallback and output.} On invalid JSON or any exception, \texttt{\_fallback\_blueprint} assigns conservative types and policies from the sniffed types, keeping the pipeline running without the LLM. The agent writes \texttt{schema\_blueprint} and \texttt{errors}; this blueprint drives every decision in Agent~3.

\section{Agent 3: Preprocessor}

\textbf{Purpose.} Agent~3 (\texttt{agent3\_preprocessor}) performs deterministic cleaning and feature construction guided entirely by the \texttt{schema\_blueprint}. It reads \texttt{\_df\_cache}, \texttt{schema\_blueprint}, and \texttt{raw\_profile}, returning early if either input is missing. The staged pipeline is shown in Figure~\ref{fig:agent3}.

\textbf{Profile selection.} \texttt{\_build\_preprocessing\_config} selects a \texttt{strict}, \texttt{balanced}, or \texttt{lenient} profile. A pinned profile is honoured; otherwise \texttt{\_detect\_dataset\_domain} inspects finance keywords and currency/percentage tags, choosing \texttt{strict} for finance/sales data. Each profile sets currency limits, a maximum tax rate, reconciliation tolerances, and quality weights.

\begin{figure}[H]
\centering
\includegraphics[width=0.6\textwidth,keepaspectratio]{agent3pic.png}
\caption{Workflow of Agent~3: Preprocessing Pipeline}
\label{fig:agent3}
\end{figure}

\textbf{Cleaning pipeline.} Processing runs as an ordered, logged sequence. Step~0 removes exact duplicates with \texttt{dedup\_exact\_rows}, asserting the count matches Agent~1 and aborting via \texttt{\_assert\_row\_survival\_or\_abort} if under half the rows survive. Step~1 (\texttt{\_clean\_currency\_values}) uses \texttt{\_normalize\_currency\_text} to convert parenthetical negatives, strip currency symbols and codes, and resolve European versus US decimals, writing a \texttt{\_parse\_failed} flag. Step~2 (\texttt{\_coerce\_types}) casts to intended types, tracking new NaNs; Step~3 standardizes text and Step~3b re-deduplicates after canonicalization.

Step~4 (\texttt{\_impute}) handles missing values per each column's \texttt{null\_policy}/\texttt{imputation\_strategy}, supporting flag-only, identifier row drop, and mean, median, mode, or unknown-label fills. Step~5 encodes categoricals; Step~6 (\texttt{\_clip\_outliers}) applies \texttt{\_adaptive\_outlier\_clipping} with percentile bounds for skewed/high-risk columns and the IQR $1.5\times$ rule otherwise; Step~7 (\texttt{\_scale\_columns}) applies Min--Max scaling while keeping \texttt{\_raw} and \texttt{\_scaled} copies in \texttt{scaling\_params}. Step~8 (\texttt{\_extract\_date\_features}) derives calendar features, and Step~9 (\texttt{\_derive\_business\_metrics}) uses whole-word matching (\texttt{\_find\_col}) and an 80\% numeric check to derive profit, margin, revenue per unit, total revenue, discount, budget variance, and total cost when possible.

\textbf{Validation and quality.} \texttt{\_validate\_count\_ranges} and \texttt{\_validate\_financial\_constraints} flag invalid counts and implausible tax rates into a \texttt{ColumnLedger}, and \texttt{\_compute\_enhanced\_quality\_score} combines remaining nulls, validation failures, and duplication into a 0--100 \texttt{data\_quality} score.

\textbf{Output and interaction.} Every step appends to \texttt{preprocessing\_log} with null diffs, and the cleaned frame is exported to \texttt{outputs/cleaned\_data.csv}. The agent updates \texttt{cleaned\_df}, \texttt{cleaned\_csv\_path}, \texttt{scaling\_params}, \texttt{preprocessing\_log}, \texttt{data\_quality}, \texttt{column\_ledger}, and reliability. Critical failures trigger \texttt{\_early\_exit\_with\_error}. The \texttt{cleaned\_df} and \texttt{scaling\_params} feed Agent~4.

\section{Agent 4: Statistical Analysis}

\textbf{Purpose.} Agent~4 (\texttt{agent4\_analysis}) turns the cleaned dataset into descriptive statistics and chart artifacts. It reads \texttt{cleaned\_df} and \texttt{schema\_blueprint}, errors out if no cleaned frame exists, clears previous PNGs with \texttt{\_clear\_chart\_dir}, and saves figures through \texttt{\_save} into \texttt{outputs/charts}. Figure~\ref{fig:agent4} outlines the analysis flow.

\textbf{Column selection.} \texttt{\_numeric\_cols} and \texttt{\_categorical\_cols} exclude validation columns (\texttt{\_parse\_failed}, \texttt{\_range\_failed}), scaling backups (\texttt{\_raw}, \texttt{\_scaled}), identifiers, datetime columns, and columns marked \texttt{analysis\_allowed = false}, so analysis runs only on meaningful fields.

\begin{figure}[H]
\centering
\includegraphics[width=0.6\textwidth,keepaspectratio]{agent4pic.png}
\caption{Workflow of Agent~4: Statistical Analysis}
\label{fig:agent4}
\end{figure}

\textbf{Analyses.} \texttt{\_descriptive\_stats} computes count, mean, median, standard deviation, variance, extremes, quartiles, skewness, and kurtosis. \texttt{\_correlation} computes Pearson and Spearman matrices, lists strong pairs with $|r| \geq 0.5$ (strong when $\geq 0.7$), and renders a heatmap. \texttt{\_growth\_rates} uses a revenue column with \texttt{\_month}/\texttt{\_year} features for month- and quarter-over-quarter charts, \texttt{\_top\_bottom\_rankings} ranks revenue across up to three categories, and \texttt{\_seasonality} plots monthly averages with best/worst months.

\texttt{\_detect\_anomalies} flags values whose absolute $z$-score exceeds 3.5:
\begin{equation}
z = \frac{x - \mu}{\sigma}
\end{equation}
where $x$ is a value and $\mu$, $\sigma$ the column mean and standard deviation. \texttt{\_category\_distributions} charts categories with at most 15 levels, and \texttt{\_regression\_trends} fits an OLS line on the \texttt{\_month} axis with \texttt{scipy.stats.linregress}, recording slope, intercept, $R^2$, $p$-value, and significance ($p < 0.05$). \texttt{\_distribution\_charts} draws a box plot and revenue histogram, and \texttt{\_derived\_metrics\_charts} plots profit margin over time and a derived-metrics summary.

\textbf{Output and interaction.} The agent writes the \texttt{stats} dictionary and \texttt{chart\_paths}, updates \texttt{errors}, and records reliability (0.9 when statistics were produced, otherwise 0.4). These statistics and charts form the evidence base consumed by the validation and reporting stages.

\chapter{RESULTS AND ANALYSIS}

\section{Observed Progress}
The current codebase already demonstrates an end-to-end pipeline from raw data ingestion through statistical analysis and chart generation. In the sample execution log, Agent 4 generates multiple chart types after profiling, semantic tagging, and preprocessing complete. That means the system is no longer just a design proposal; it is now a working backend prototype up to Agent 4.

That progress is important because it confirms the proposal is executable rather than purely conceptual. The pipeline is already performing the core work the proposal promised: profiling, semantic tagging, preprocessing, and statistical analysis. The remaining validation and reporting stages are still to be built, which is appropriate for a mid-term checkpoint.

\section{Practical Output}
The most visible outputs produced by the current implementation are:
\begin{itemize}
		\item a structural profile of the uploaded dataset,
		\item a semantic schema blueprint,
		\item a cleaned dataset with quality metrics and an audit trail,
		\item statistical summaries and charts,
		\item per-stage reliability metadata with confidence scores.
\end{itemize}

		These outputs are useful because they show both machine-readable state and human-readable artifacts. The system can therefore be inspected by later agents as well as by project evaluators. That dual nature is one of the most practical strengths of the current implementation.

\section{Analysis}
The strongest result so far is that the system separates concerns well. Agent 1 does not try to clean data, Agent 3 does not try to infer semantics, and Agent 4 focuses only on analysis rather than cleaning. That separation makes the pipeline easier to maintain and easier to explain in a report. Another important result is the reliability mechanism: the system keeps track of confidence and decision readiness instead of assuming every stage succeeds equally well.

From an engineering perspective, the current codebase is already behaving like a real analytics pipeline. It does not just transform rows and columns; it builds a story about the dataset in stages. It starts with raw structure, adds meaning, cleans the data, and computes evidence in the form of statistics and charts. The later stages that check that evidence and produce a report are planned as the next steps. That staged sequence is the core technical value of the project.

The depth of the implementation is also visible in the way the backend handles data quality. Rather than relying on one generic cleaning rule, the pipeline uses multiple checks and logs to capture what changed and why. That makes the output more defensible and more appropriate for business datasets where context matters.

\section{Current Challenges}
The main remaining work spans both new functionality and refinement. The two unimplemented agents (validation and reporting) still need to be built, and the pipeline also needs broader testing on different dataset shapes and a clearer evaluation section for the final report. The planned validation gate will need careful tuning so that it remains strict without rejecting acceptable outputs unnecessarily.

Another challenge is balancing flexibility and consistency. The system needs to handle varied datasets, but it also needs to produce stable and repeatable results. That balance is important because the final project should be practical without becoming unpredictable.

At this stage, the project is best described as functional but still maturing. The core behavior is already present, but the last phase needs stronger evaluation, documentation, and presentation work before submission.

\chapter{REMAINING TASKS}

\begin{enumerate}
		\item Implement Agent~5 (Validation and Quality Guardrail) to check the statistical outputs, charts, and quality metrics and to gate low-confidence results before reporting.
		\item Implement Agent~6 (Report Generation) to assemble the validated statistics and charts into a self-contained final report.
		\item Improve the presentation layer and decide whether a user-facing interface should be included in the final deliverable.
		\item Expand automated tests across preprocessing and the new validation and reporting stages.
		\item Prepare the final evaluation section with sample outputs and documented observations, then clean up the report language, citations, and formatting before final submission.

In addition to these tasks, the team needs to keep the report aligned with the actual backend behavior. As the codebase evolves, the document should continue to describe what the system really does rather than only what was originally planned. That alignment will matter in the final defense because it keeps the report credible and technically consistent.

\chapter{CONCLUSION}
The project has reached a strong mid-point. The core backend pipeline is implemented through Agent~4, the agents are coordinated through shared state, and the system can process data through profiling, semantic tagging, preprocessing, and statistical analysis. Validation (Agent~5) and report generation (Agent~6) remain to be implemented, so the remaining work combines this new functionality with refinement of the existing stages. This gives the project a solid base for final evaluation and presentation.

At this stage, the project can already be described as a functioning multi-agent analytics backend up to the statistical analysis stage. Its main value lies in combining deterministic processing with controlled language-model assistance while keeping the workflow auditable. The next phase is to build the validation and reporting agents and to strengthen the presentation, test coverage, and evaluation so that the final report can present the system confidently and with evidence.
